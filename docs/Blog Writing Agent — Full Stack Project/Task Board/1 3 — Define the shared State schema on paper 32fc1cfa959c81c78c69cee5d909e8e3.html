Overview

Before writing any LangGraph code, design the complete state object that flows through every node in the agent graph. This is the single most important design decision in the project — every node reads from and writes to this object.

Why This Matters

Changing the state schema mid-development forces updates across every node. Getting it right upfront saves enormous refactoring effort. Think of it as your database schema — plan it carefully.

The State Fields

Design your TypedDict state with these fields:

  • prompt (str): The original user input, never modified
  • needs_research (bool): Set by the Router node
  • research_queries (list[str]): Generated search queries
  • research_results (list[dict]): Raw tool outputs from web search and fetch
  • research_summary (str): Compressed, clean summary from the Summarizer node
  • blog_plan (dict): Structured plan with title, sections, tone, audience
  • tasks (list[dict]): One task per section with description and word count
  • task_outputs (dict): Maps task ID to written section content
  • draft (str): Full assembled markdown blog post
  • final_post (str): Edited final version
  • human_feedback (str | None): Feedback injected at HIL checkpoints
  • iteration_count (int): Tracks revision loops, capped at 3
  • thread_id (str): UUID for this specific run
  • status (str): Current state — 'running', 'awaiting_human', 'complete', 'error'
  • interrupt_type (str | None): 'plan_review' or 'final_review'
  • error (str | None): Error message if something fails
  • metadata (dict): tone, target_audience, word_count_target, seo_keywords

Steps

  1. Write out all fields on paper or in a notes doc first
  1. Create backend/agent/state.py
  1. Import TypedDict from typing and Annotated for fields that use reducers
  1. Define the BlogAgentState class with all fields and type hints
  1. Add inline comments explaining each field's purpose and which node sets it
  1. Review: can every node accomplish its task using only what's in state?

Acceptance Criteria

  • state.py file exists with a complete TypedDict definition
  • Every field has a type hint and a comment
  • No node will need data that isn't in the state

Notes

For task_outputs, use Annotated[dict, operator.or_] as the reducer if you later implement parallel task execution — this allows parallel nodes to merge their outputs into the dict without overwriting each other.
