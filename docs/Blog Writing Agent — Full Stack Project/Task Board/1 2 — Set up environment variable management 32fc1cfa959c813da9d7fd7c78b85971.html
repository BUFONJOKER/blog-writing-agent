1.2 — Set up environment variable management

Est. Hours1
PhasePhase 1 - Setup
PriorityHigh
SprintSprint 1
StatusNot Started

Overview

Create .env.example files in each service subfolder that document every environment variable the application needs. These example files are committed to git; actual .env files are never committed.

Why This Matters

Missing environment variables cause runtime crashes that are hard to debug. Having explicit .env.example files acts as living documentation and makes onboarding new developers (or yourself after a break) much faster.

Steps

  1. In mcp_server/, create .env.example with:
    • TAVILY_API_KEY=your_tavily_key_here
    • MCP_AUTH_TOKEN=your_secret_token_here
    • PORT=8001
  1. In backend/, create .env.example with:
    • OPENAI_API_KEY=your_openai_key_here
    • ANTHROPIC_API_KEY=your_anthropic_key_here
    • MCP_SERVER_URL=http://localhost:8001
    • MCP_AUTH_TOKEN=your_secret_token_here
    • DATABASE_URL=postgresql://user:password@localhost:5432/blogagent
    • LANGCHAIN_TRACING_V2=true
    • LANGCHAIN_API_KEY=your_langsmith_key_here
    • SECRET_KEY=your_jwt_secret_here
  1. In frontend/, create .env.example with:
    • NEXT_PUBLIC_BACKEND_URL=http://localhost:8000
  1. Add a comment at the top of each file: # Copy this file to .env and fill in your values. Never commit .env
  1. Confirm .gitignore has *.env and .env entries

Acceptance Criteria

  • Three .env.example files exist, one per service
  • Every variable used anywhere in the codebase is documented in the relevant example file
  • Running grep -r 'os.getenv' backend/ shows no undocumented variables

Notes

Use python-dotenv in the backend to load .env automatically. In production, set env vars directly in the hosting platform (Railway, Render, Vercel) rather than using .env files.
