====================================================================
1) PROJECT OVERVIEW
====================================================================
This is a full-stack AI blog generation system with three primary services:
1. Frontend (Next.js + React + TypeScript)
2. Backend (FastAPI + LangGraph + LangChain + PostgreSQL persistence)
3. MCP tool server (FastMCP + Tavily search/extract)

Primary user flow:
- User submits prompt in frontend chat UI.
- Backend starts a LangGraph workflow with the prompt.
- Workflow optionally performs external research through MCP tools.
- Workflow plans, drafts, assembles, edits, critiques, and finalizes a blog.
- If human review is required, workflow pauses and awaits approve/reject.
- Final post is stored in DB and returned/streamed to frontend.

Known current project constraints (from docs):
- Runtime can be long for fully researched runs.
- Token usage is high in current model setup.
- Post-generation chat-edit experience can be expanded further.

Routing logic highlights:
- START -> router_node
- router_node decides:
  - needs_research=True -> research_query_gen_node
  - False -> summarizer_node
- researcher_node conditionally loops through researcher_tools based on tool calls.
- research_loop controls return-to-research vs planner progression.
- critic_node -> human_review
- human_review routes:
  - approved -> finalize_node
  - rejected and revision_cycles < 3 -> task_executer_node
  - otherwise -> finalize_node
- finalize_node -> END

3.5) Node-by-Node Behavior
Files: backend/agent/nodes/*.py

- router.py
  - LLM structured decision: needs_research (bool), topic.
  - Conservative policy: if unsure, favors research.

- research_query_gen.py
  - Generates 3-5 focused search queries.
  - Outputs research_gaps and more_research_needed.

- researcher.py
  - Binds MCP tools to model.
  - Reconstructs/merges historical tool results.
  - Deduplicates by URL.
  - Stops on futility patterns (future/not available language).
  - Enforces tool-call limit guard.
  - Returns summary snapshot + loop flags.

- summarizer.py
  - If research is enabled, condenses research output.
  - If not, generates summary from internal model knowledge.
  - Ensures non-empty fallback summary.

- research_loop.py
  - Decides continue research vs planning.
  - Hard stops if tool_call_count >= max_tool_calls.

- planner.py
  - Produces structured BlogPlan:
    - title, subtitle, tone, audience
    - sections with key_points, keywords, word targets
    - ordered tasks list
  - Repairs tasks/sections mismatch if needed.

- task_executer.py
  - Drafts section content for each planned task.
  - Revision mode supports critic-feedback-based rewrite.
  - Maintains markdown structure and SEO constraints.

- assembler.py
  - Combines section outputs to structured BlogDraft schema.
  - Produces draft + meta_description + slug + keyword usage + title.

- editor.py
  - Final pass to improve coherence, readability, transitions.
  - Avoids adding new factual claims.

- critic.py
  - Structured critique: issues/suggestions + score + confidence.
  - Sets needs_revision boolean for loop control.

- finalize.py
  - Final polish on edited draft.
  - Returns final_post markdown.

Route behavior details:

- stream.py

  - Terminates on {"event":"stream_end"}.
- status/final_post/user_posts/delete_chat
  - Read/write run and output state with ownership checks.

5.2) MCP Server Bootstrap
File: mcp_server/server.py
- Creates FastMCP("blog-research-tools").
- Registers two tools:
  - web_search_tool
  - fetch_page_tool
- Transport modes:
  - default: stdio (local usage)
  - --sse: host 0.0.0.0, port from PORT env (default 8000)

5.3) Tool Config
File: mcp_server/config.py
- Loads .env.
- Exposes TAVILY_API_KEY.

5.4) Tool: web_search_tool
File: mcp_server/tools/web_search.py
- Inputs:
  - query (required)
  - max_results (1-10, clamped)
- Calls Tavily search.
- Returns JSON string with query + normalized results list:
  - url, title, content, score

5.5) Tool: fetch_page_tool
File: mcp_server/tools/fetch_page.py
- Inputs:
  - url (required)
  - query (optional focus)
- Calls Tavily extract (advanced depth).
- Merges top chunks, cleans content (unicode/junk stripping), truncates by tokens.
- Returns structured dict with:
  - tool, title, content, snippet, source, score, word_count

5.6) Containerization
File: mcp_server/Dockerfile
- Multi-stage build with uv dependency sync.
- Runtime image based on python:3.11-slim-bookworm.
- Healthcheck using pgrep for server.py process.
- Default container command runs SSE mode.

9) END-TO-END FLOW TRANSCRIPT (OPERATIONAL)
====================================================================
1. User logs in from frontend auth page.
2. Frontend stores auth identity in localStorage and uses cookie session for API calls.
3. User submits prompt in BottomComposer.
4. Frontend calls POST /blog/generate.
5. Backend starts async workflow and returns thread_id + stream_url.
6. Frontend opens EventSource to /blog/stream/{thread_id}.
7. Backend publishes node updates while graph runs.
8. Frontend visualizes node progress and node outputs.
9. If critic requires review, backend marks waiting_approval and stream sends waiting event.
10. User approves/rejects via frontend.
11. Frontend sends POST /blog/review.
12. Backend resumes graph from human_review checkpoint.
13. Backend finalizes output, persists final markdown + metadata.
14. Backend sends completed event and closes stream.
15. Frontend fetches final post and renders markdown output.
16. User can revisit prior threads via sidebar history.
